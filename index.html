Слайд 2 проблема

Проблема: ноутбуки сложно проходить самостоятельно

  • 4 ноутбука: основы Inspect AI, статистика евалов, LLM-судья, агенты и tool use
  • Фидбек студентов курса: непонятные формулировки заданий, зависающие евалы на слабом железе, проверки, которые проходят у неверных решений
  • Цель: чтобы ноутбуки можно было проходить самостоятельно, без семинара и помощи ментора
Устно: сами материалы полные, проблемы именно с самостоятельным прохождением. Сюда хорошо ложится реальная цитата из анкет или чата семинаров.

Слайд 5 направление 2 · Technical/Runtime

Надёжность запуска

  • Самая частая жалоба: евалы зависают. Добавлены лимиты в ноутбуках 3-4 (max_connections, timeout, time_limit) и FAQ в README о том, какая настройка за что отвечает (PR #30)
  • Дефолтная модель заменена: устаревшая llama2 → llama3.2:3b, где надёжен tool-calling — qwen2.5:3b (PR #29)
  • Датасет MMLU получил локальный parquet-fallback: ноутбук работает без доступа к HuggingFace (PR #15)
  • Онбординг: матрица провайдеров, настройка .env, гайд по сохранению виджетов, мини-гайд «что делать, если тул завис»
  • Colab: бейджи «Open in Colab» и полная адаптация, ноутбуки проходятся в бесплатном Colab, Ollama поднимается прямо внутри VM
Устно: для масштаба: на CPU без GPU одна генерация 3b-модели занимает 25-30 секунд, ReAct-цикл несколько минут на сэмпл. Без лимитов не отличить зависание от долгого ответа.
Слайд 6 направление 3 · Correctness/QA

Корректность проверок

  • Свой CI: check_notebooks.py, ~310 строк офлайн-проверок (валидность ipynb, компиляция ячеек с учётом placeholder'ов, разрешение импортов против дрейфа API, битые ссылки, нумерация). Гоняется на каждый PR (PR #3)
  • Судья в ноутбуке 3 засчитывал фразу «I believe...» как грейд I: regex с опциональным префиксом. Ужесточён до обязательного GRADE: (PR #35)
  • Assignment 3 (position bias) сравнивал прогоны с разным контентом: конфаундер. Пул вопросов заморожен, варьируется только позиция ответа (PR #4)
  • Тесты статистики ужесточены: наивная формула больше не проходит, привязка к формуле из статьи (PR #33, #38)
  • Bugfix week 4: match(numeric=True) заменён на точное сравнение
Устно: самый понятный пример для показа: grade_pattern, где из-за одной строки regex фраза судьи «I believe...» засчитывалась как оценка I.
Слайд 7 эксперименты

Изменения проверялись экспериментами

  • Правило команды: ни одного утверждения без запускаемого репро-скрипта
  • Смена модели по итогам серии экспериментов E0–E5: llama3.2:3b vs 1b на MMLU, n=60, gap +0.20, t=3.23, p=0.002. Тот же прогон на n=15 разницы не показывает; этот же приём используется в ноутбуке 2 для демонстрации статистической мощности
  • llama3.2:3b провалила tool-calling (галлюцинирует операнды, не зовёт submit, 0.0 acc против 0.4 без тулов), qwen2.5:3b справилась: 0.8 acc, ноль зависаний. Поэтому в ноутбуке 4 другая модель
  • Аудит LLM-судьи на сбалансированном срезе: маленькая модель соблюдает формат GRADE: C/I, но сама оценка не связана с её же рассуждением
Устно: завершить тем, что методологию из самих ноутбуков (сравнение моделей, аудит судьи) команда применяла к решениям по этим же ноутбукам.

Слайд 10 финал

Чему научились

  • Тест должен падать на наивном решении, иначе это не тест. Round-trip проверки недостаточно, нужна привязка к эталону
  • Соблюдение формата не означает осмысленной оценки: маленькая модель стабильно пишет GRADE:, но грейдит наугад. Судью нужно проверять на сбалансированном срезе
  • Самый дешёвый эксперимент, который может закрыть вопрос, стоит делать первым
  • «Нет эффекта» при малой выборке часто означает лишь нехватку данных: n=15 против n=60 на одной паре моделей
