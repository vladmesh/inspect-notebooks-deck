Доработка учебных ноутбуков Inspect AI

Устно: одна фраза о том, почему евалы важнее для AI Safety, чем для capabilities: прежде чем доверять модели, нужно уметь её измерять. Наши ноутбуки учат ровно этому.
Слайд 2 проблема

Хорошие материалы, через которые тяжело пройти

  • 4 ноутбука: основы Inspect AI → статистика евалов → LLM-судья → агенты и tool use
  • Фидбек студентов курса: непонятные формулировки заданий, зависающие евалы на слабом железе, проверки, которые проходят у неверных решений
  • Цель: довести ноутбуки до уровня «человек проходит сам, без семинара и без ментора рядом»
Устно: подчеркнуть, что материал контент-полный, но UX обучения хромал. Хороший ход: одна реальная цитата из анкет или чата семинаров, если найдётся.

  • Добавлены недостающие объяснения: солверы и CoT multiple-choice, ReAct vs просто вызов тулов, base vs instruction-tuned модели, глоссарий к статистике
  • Новое вводное: technical overview Inspect AI перед первым запуском, предисловие в README
Устно: взять один пример «было/стало», например формулировку 4.1 до и после. Показать скриншотом, не пересказом.
Слайд 5 ось 2 · Technical/Runtime

Надёжность: евал доезжает до конца на любом железе

  • Главная жалоба: евалы зависают. Ответ: runtime limits в ноутбуках 3-4 (max_connections, timeout, time_limit) + FAQ в README о том, какая ручка от чего (PR #30)
  • Дефолтная модель заменена: устаревшая llama2 → llama3.2:3b, где надёжен tool-calling — qwen2.5:3b (PR #29)
  • Датасет MMLU получил локальный parquet-fallback: ноутбук работает без доступа к HuggingFace (PR #15)
  • Онбординг: матрица провайдеров, настройка .env, гайд по сохранению виджетов, мини-гайд «что делать, если тул завис»
  • Colab: бейджи «Open in Colab» и полная адаптация, ноутбуки проходятся в бесплатном Colab, Ollama поднимается прямо внутри VM
Устно: дать масштаб цифрой: на CPU без GPU одна генерация 3b-модели идёт 25-30 секунд, ReAct-цикл — минуты на сэмпл. Без лимитов «завис» и «долго думает» неразличимы, поэтому лимиты обязательны.
Слайд 6 ось 3 · Correctness/QA

Корректность: проверки ловят неверные решения, а не пропускают их

  • Свой CI: check_notebooks.py, ~310 строк офлайн-проверок (валидность ipynb, компиляция ячеек с учётом placeholder'ов, разрешение импортов против дрейфа API, битые ссылки, нумерация). Гоняется на каждый PR (PR #3)
  • Судья в ноутбуке 3 засчитывал фразу «I believe...» как грейд I: regex с опциональным префиксом. Ужесточён до обязательного GRADE: (PR #35)
  • Assignment 3 (position bias) сравнивал прогоны с разным контентом: конфаундер. Пул вопросов заморожен, варьируется только позиция ответа (PR #4)
  • Тесты статистики ужесточены: наивная формула больше не проходит, привязка к формуле из статьи (PR #33, #38)
  • Bugfix week 4: match(numeric=True) заменён на точное сравнение
Устно: лучший пример для показа — grade_pattern: одна строка regex, из-за которой судья «оценивал» собственные вводные слова. Наглядно и смешно.
Слайд 7 кульминация

Не «поправили текст», а проверили экспериментами

  • Правило команды: ни одного утверждения без запускаемого репро-скрипта
  • Смена модели — по итогам серии экспериментов E0–E5: llama3.2:3b vs 1b на MMLU, n=60, gap +0.20, t=3.23, p=0.002. Тот же прогон на n=15 разницы не видит: живая демонстрация статмощности из ноутбука 2
  • llama3.2:3b провалила tool-calling (галлюцинирует операнды, не зовёт submit, 0.0 acc против 0.4 без тулов), qwen2.5:3b справилась: 0.8 acc, ноль зависаний. Поэтому в ноутбуке 4 другая модель
  • Аудит LLM-судьи на сбалансированном срезе: маленькая модель держит формат GRADE: C/I, но грейд не связан с её же рассуждением. Формат ≠ суждение
Устно: финальная мысль слайда, она же рекурсия всей защиты: мы применили методологию из ноутбуков к самим ноутбукам. Учебник про евалы прошёл собственный евал.

Слайд 10 финал

Чему научились

  • Тест должен падать на наивном решении, иначе это не тест. Round-trip проверки недостаточно, нужна привязка к эталону
  • Формат ≠ суждение: маленькая модель безупречно пишет GRADE: и при этом грейдит наугад. Судью нужно аудитить на сбалансированном срезе
  • Самый дешёвый эксперимент, способный убить идею, идёт первым
  • «Нет эффекта» при малом n — артефакт, а не результат: n=15 против n=60 на одной паре моделей
